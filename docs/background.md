## 基于 LLM 的 Markdown 文本批量翻译

### 一、项目简介

本项目用于对大段 Markdown 文档进行自动化翻译，力图保留原有的 Markdown 结构。与一般翻译流程不同的是，本项目先对文本进行长文本摘要，再在实际翻译时将摘要内容作为上下文，帮助大语言模型（LLM）更好地理解文本背景并进行准确翻译。

#### 项目主要流程

- **读取并解析 Markdown 文件**
- **对全文按「约 2 万词」进行首轮分段**（如果原文小于 2 万词，则无需拆分），并对每段进行整体概要（摘要）。
- **再次细分翻译分段**：对原文（或者分段后的每部分）再按「约 2,000 词」进行二次分段，用于实际调用 LLM 翻译。在翻译时，将对应的摘要内容与 2,000 词分段的文本一并传入 LLM，帮助模型理解上下文。
- **将翻译结果拼接回原有的 Markdown 格式**，形成最终输出。

#### 示例

- **2 万词内**：不用拆分，先整体做摘要，再按 2,000 词翻译分段，翻译时带上整体摘要。  
- **4 万词**：先拆分为两段，每段 2 万词；对每段做摘要，然后各自再分割成若干个 2,000 词，翻译时带上对应的 2 万词摘要。  
- **6 万词**：分为三段，每段 2 万词；对每段做摘要，翻译时各段分别带其对应的摘要。

#### 单词计数说明

- **字 (character)**：字符数  
- **词 (word)**：单词数  
- **英文文本**：按照标准英文单词进行计数（包括带连字符和数字的单词）  
- **中文文本**：每个汉字视为一个词进行计数  
- **混合文本**：同时支持中英文混合计数  
- **特殊处理**：  
  - 代码块内容不计入单词统计，避免影响分段  
  - 保持 Markdown 格式标记的完整性  
  - 优先在自然段落边界进行分段

### 三、分段与翻译核心思路

#### 1. 第一轮分段（约 2 万词）并摘要

- 读取整个 Markdown 文本，统计词数。
- 若小于 2 万词，则无需拆分，直接对全文做摘要。
- 若超过 2 万词，每 2 万词左右切分一次（推荐以自然段边界切分）。
- 对每个分段调用 LLM 生成段落级摘要。

#### 2. 第二轮分段（约 2,000 词）并翻译

- 对原文或对应的首轮分段再次进行精细切分，每次约 2,000 词左右。
- 每个 2,000 词分段翻译时，将「对应的摘要」和「分段文本」一起作为 Prompt 发给 LLM，以便模型在理解上下文的基础上进行翻译。
- 将翻译结果记录下来，并与原文段落/块建立对应关系。

#### 3. 拼接并保持 Markdown 结构

- 最终将所有翻译后的文本按原顺序拼接，保留标题、列表、代码块等 Markdown 结构。
- 输出完整的翻译后 Markdown 文件。

### 五、中级翻译流程：翻译 + 自动评分 + 润色

在上述基础功能之上，你还可以引入「自动评分与润色」的中级翻译流程。具体思路如下：

1. **初次翻译**：先对每个 2,000 词小块进行一次常规翻译。
2. **自动评分**：对比「原文」与「译文」，由 LLM 生成「译文评分」。评分可从以下五个维度（示例）给出 0~10 分：
   - 准确性 (Accuracy)
   - 完整性 (Completeness)
   - 流畅性 (Fluency)
   - 术语与用词 (Terminology & Word Choice)
   - 风格与语气 (Style & Tone)

   总分即上述五个维度的合计（满分 50 分）。若总分低于 45，则需要将「原文、译文、评分结果」交给后续模型进行**润色或重译**。

3. **润色/重译**：根据评分结果，若得分较低，可以让 LLM 根据评分中的问题，对译文进行二次修订。

以上示例展示了核心逻辑：

1. **结构化评分**（JSON）
2. **解析分数**
3. **if / else 分支处理**，对不合格译文做润色或重译

在大规模分段翻译中，你可以针对每个 2,000 词小块的译文做评分，若不合格再润色；最终再把所有通过评分或润色后的内容合并。此流程能显著提高整体翻译质量，但也会带来额外的 API 调用成本和时间开销。

### 六、后续改进与注意事项

- **摘要策略**：可根据文档类型、语言特点进一步优化摘要提示。
- **Markdown 解析**：若对结构保留要求较高，需使用 Markdown AST 解析，再分段处理，以防断行、代码块等格式混乱。
- **容错与重试机制**：网络调用、超时以及模型返回异常等情况需要结合实际使用场景做好重试或异常处理。
- **自动评分与润色**：
  - 建议使用 JSON 结构化评分，方便代码自动解析
  - 可以根据评分结果做多轮润色
  - 也可将评分日志写入数据库或文件，便于后续质量分析
- **成本与性能**：自动评分和润色会增加额外的模型调用次数，需要在质量与成本之间做平衡。

以上即为本项目的使用说明，以及在「翻译+自动评分+润色」流程上的扩展思路。希望能帮助你在批量翻译 Markdown 文档时，实现更高质量、更自动化的处理体验。
