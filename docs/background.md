## 基于 LLM 的 Markdown 文本批量翻译

### 一、项目简介

本项目用于对大段 Markdown 文档进行自动化翻译，力图保留原有的 Markdown 结构。与一般翻译流程不同的是，本项目先对文本进行长文本摘要，再在实际翻译时将摘要内容作为上下文，帮助大语言模型（LLM）更好地理解文本背景并进行准确翻译。

#### 项目主要流程

- **读取并解析 Markdown 文件**
- **对全文按「约 2 万词」进行首轮分段**（如果原文小于 2 万词，则无需拆分），并对每段进行整体概要（摘要）。
- **再次细分翻译分段**：对原文（或者分段后的每部分）再按「约 2,000 词」进行二次分段，用于实际调用 LLM 翻译。在翻译时，将对应的摘要内容与 2,000 词分段的文本一并传入 LLM，帮助模型理解上下文。
- **将翻译结果拼接回原有的 Markdown 格式**，形成最终输出。

#### 示例

- **2 万词内**：不用拆分，先整体做摘要，再按 2,000 词翻译分段，翻译时带上整体摘要。  
- **4 万词**：先拆分为两段，每段 2 万词；对每段做摘要，然后各自再分割成若干个 2,000 词，翻译时带上对应的 2 万词摘要。  
- **6 万词**：分为三段，每段 2 万词；对每段做摘要，翻译时各段分别带其对应的摘要。

#### 单词计数说明

- **字 (character)**：字符数  
- **词 (word)**：单词数  
- **英文文本**：按照标准英文单词进行计数（包括带连字符和数字的单词）  
- **中文文本**：每个汉字视为一个词进行计数  
- **混合文本**：同时支持中英文混合计数  
- **特殊处理**：  
  - 代码块内容不计入单词统计，避免影响分段  
  - 保持 Markdown 格式标记的完整性  
  - 优先在自然段落边界进行分段

### 二、技术选型

- **语言**：Python（建议版本 3.7+）
- **依赖库**：
  - `openai`（或自定义封装，用于调用 Fireworks.ai 等 LLM 服务）

### 三、分段与翻译核心思路

#### 1. 第一轮分段（约 2 万词）并摘要

- 读取整个 Markdown 文本，统计词数。
- 若小于 2 万词，则无需拆分，直接对全文做摘要。
- 若超过 2 万词，每 2 万词左右切分一次（推荐以自然段边界切分）。
- 对每个分段调用 LLM 生成段落级摘要。

#### 2. 第二轮分段（约 2,000 词）并翻译

- 对原文或对应的首轮分段再次进行精细切分，每次约 2,000 词左右。
- 每个 2,000 词分段翻译时，将「对应的摘要」和「分段文本」一起作为 Prompt 发给 LLM，以便模型在理解上下文的基础上进行翻译。
- 将翻译结果记录下来，并与原文段落/块建立对应关系。

#### 3. 拼接并保持 Markdown 结构

- 最终将所有翻译后的文本按原顺序拼接，保留标题、列表、代码块等 Markdown 结构。
- 输出完整的翻译后 Markdown 文件。

### 五、后续改进与注意事项

- **摘要策略**：可根据文档类型、语言特点进一步优化摘要提示。
- **Markdown 解析**：若对结构保留要求较高，需使用 Markdown AST 解析，再分段处理，以防断行、代码块等格式混乱。
- **容错与重试机制**：网络调用、超时以及模型返回异常等情况需要结合实际使用场景做好重试或异常处理。
- **自动评分与润色**：
  - 建议使用 JSON 结构化评分，方便代码自动解析
  - 可以根据评分结果做多轮润色
  - 也可将评分日志写入数据库或文件，便于后续质量分析
- **成本与性能**：自动评分和润色会增加额外的模型调用次数，需要在质量与成本之间做平衡。

以上即为本项目的使用说明，以及在「翻译+自动评分+润色」流程上的扩展思路。希望能帮助你在批量翻译 Markdown 文档时，实现更高质量、更自动化的处理体验。

### 增加一个「高级模式」(advanced) 翻译流程

该模式下的主要思路是：
- **步骤 1**: 先进行翻译
- **步骤 2**: 基于「原文 + 译文」让模型点评可优化之处（即指出具体问题或改进点）
- **步骤 3**: 基于模型给出的可优化点，对译文进行二次润色

与传统的“评分后再润色”方案不同的是：
- 这里不再输出“评分”，而是直接给出「改进意见」。
- 随后根据这些意见对译文进行二次润色，以获得更高质量的最终译文。

与原有的流程保持一致，只是将评分+润色的环节替换为点评+润色，核心多了一个“模型点评可优化之处”的步骤。

#### 整体流程

1. **分块**（与原逻辑相同）
2. **翻译**
3. **点评**: 原文 + 译文 -> 模型 -> 提供改进点
4. **润色**: 译文 + 改进点 -> 模型 -> 生成优化后的译文

---

#### 总结与注意事项

##### 1. 点评与润色
- **高级模式的关键**在于“点评”环节要能明确指出具体改进点（比如名词术语、语序、语气、风格等细节），而不是简单给个数值评分。
- 在润色环节可让 LLM 按这些改进点逐一修订，从而得到更精准的结果。

##### 2. Prompt 设计
- 可使用更精细或更结构化的输出（如 JSON），便于程序自动识别「问题类型」并进行更有针对性的二次改写。
- 也可结合已有的「评分 + 改进点」思路，将「评分」和「改进意见」同时输出。

##### 3. 调用开销与性能
- 多次调用 LLM（翻译、点评、润色）会增加调用次数与费用，需要在成本与效果之间做平衡。
- 大段文本需要分块调用，需注意在分块之间保持风格一致性；可在最终拼接完后再做一次全局润色（可选）。

##### 4. 日志与调试
- 建议在日志中记录每段的原文、初译文、点评、润色后译文，以便后续回溯和质量分析。


## 示例：Mermaid 流程图

下面给出一个示例的 Mermaid 流程图，展示了从读取文档到分段、摘要、翻译以及可选的自动评分与润色的整体逻辑示意。你可以根据实际需要自行调整节点或增加细节。

```mermaid
flowchart TD
    A([开始]) --> B[读取并解析<br>Markdown 文件]
    B --> C[统计<br>总词数]
    C --> D{是否小于<br>2 万词？}

    D -- 是 --> E[为全文<br>生成摘要]
    E --> F[将全文细分为<br>约 2,000 词的小段]
    F --> G[调用 LLM 翻译<br>(附带全文摘要)]
    G --> H{是否执行<br>自动评分？}

    H -- 否 --> I[直接合并<br>并保持 MD 结构]
    H -- 是 --> J[对译文进行<br>自动评分]
    J --> K{得分 >= 45？}
    K -- 是 --> I
    K -- 否 --> L[润色或重译<br>(可多次迭代)]
    L --> J

    D -- 否 --> M[按约 2 万词<br>分割为多段]
    M --> N[分别为每段<br>生成概要摘要]
    N --> O[对每段再细分<br>约 2,000 词]
    O --> P[调用 LLM 翻译<br>(附带对应分段摘要)]
    P --> Q{是否执行<br>自动评分？}

    Q -- 否 --> R[合并译文<br>保持 MD 结构]
    Q -- 是 --> S[对译文进行<br>自动评分]
    S --> T{得分 >= 45？}
    T -- 是 --> R
    T -- 否 --> U[润色或重译<br>(可多次迭代)]
    U --> S

    I --> V([输出最终翻译])
    R --> V
```

### 图示说明

- **读取并解析**：先解析 Markdown 文件，统计单词数。
- **判定是否需「首轮大分段」**：以 2 万词为阈值判断是否需要大分段。
- **生成摘要**：
  - 若无需大分段则为全文生成摘要。
  - 若需要大分段则分别对每个 2 万词段生成对应摘要。
- **二次分段并翻译**：
  - 将上一步的文本（或每个大段）细分为约 2,000 词的小段。
  - 以摘要作为上下文调用 LLM 翻译。
- **自动评分和润色（可选）**：
  - 可对译文进行自动评分。
  - 不达标的部分可再次润色或重译，支持多次迭代。
- **合并输出**：最终将所有翻译后的文本合并回原 Markdown 结构，输出完整文档。
